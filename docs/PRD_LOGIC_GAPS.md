## §3.2 预测周期管理（7个问题）

🔴 **1. 周期的"当前周期"如何确定** ✅已确认
系统里可能有多个历史周期（2025FC2、2026FC1），Dashboard 和填报页面显示哪个？有没有"当前激活周期"的概念，由谁设置（Admin手动？还是系统取最新的）？
→ **确认结论：按时间窗口自动锁定**。FillTimeStart/FillTimeEnd 时间窗口不重叠，同一时间只有唯一周期有效。系统按当前时间自动取落在窗口内的周期，无需IsActive字段。

🔴 **2. 填报窗口的时间校验边界** ✅已确认
填报截止判断用的是服务器时间还是浏览器本地时间？客户端和服务器时区差会不会导致本应截止的记录被错误提交？
→ **确认结论：统一用服务器时间（DateTime.UtcNow）**，不用浏览器时间。

🟢 **3. 预测月份范围的边界情况**
`period_start = 2026-01，period_end = 2026-01`，是1个月还是2个月？边界定义需要明确。

🔴 **4. 延期窗口和正常窗口的关系** ✅已确认
延期窗口是正常窗口的严格延长期（只有白名单可提交）？还是两个独立时间段？无延期信息时默认行为是什么？
→ **确认结论：两个独立窗口**。FillTimeEnd 锁定后，ExtensionStart 可灵活设定（=FillTimeEnd 或更晚），Extension窗口独立存在。ExtensionEnd = ExtensionStart + N天。

🔴 **5. 周期创建后能否修改字段** ✅已确认
周期已开始填报，Admin 能改哪些字段（名称？截止时间？）？改截止时间对已提交数据有什么影响？
→ **确认结论：Admin可改任意字段，但有数据的周期禁止删除**。底线：已有数据的周期不能删，字段可改。

🔴 **6. 周期结束后能否重新打开** ✅已确认
周期过了截止时间+延期时间，还能重新激活吗？数据怎么处理？
→ **确认结论：FillTimeEnd 过期后锁定，不可改。Extension 窗口可继续调整（通过调 ExtensionEnd 实现"延期重开"）。FillTimeEnd 本身不能改，只能用延期窗口来处理特殊情况。**

🟢 **7. 周期删除的限制** ✅已确认（隐含在Q5中）
如果周期已有预测数据，Admin 能删除周期吗？删除后数据怎么办？
→ **确认结论：底线——已有填报数据的周期禁止删除。字段可改，删除是禁区。**

## §3.3 填报流程（8个问题）

🔴 **8. 首次进入填报页面的默认状态** ✅已确认
首次进入时是空白表单？还是系统自动列出品牌匹配的客户供选择？有没有快速批量导入模式？
→ **确认结论：空白列表。显示已填报条目的数量和金额汇总。无数据时全空白。**

🔴 **9. 填报数据的自动保存** ✅已确认
如果销售填了很久浏览器崩溃，数据会不会丢？需不需要服务器端AutoSave？
→ **确认结论：不做 AutoSave。依赖模板上传/下载。**

🔴 **10. 草稿和正式提交的数据是否在同一个表** ✅已确认（补充确认）
Dashboard统计时包不包括草稿？审批流的数据口径怎么算？
→ **确认结论：同一表，Status区分。Dashboard和报表只含已审批通过数据，不含过程数据。**

🔴 **11. 填报时产品与客户的联动逻辑** ✅已确认
选了客户A后，产品下拉框是显示所有产品、品牌对应产品、还是该客户历史填报过的产品？
→ **确认结论：无联动，产品全部开放，选任意客户后产品下拉显示全部产品。**

🔴 **12. 填报月份列的展示逻辑** ✅已确认
月份列是横排（每月一列）还是竖排（一个月一行）？12个月是12列？
→ **确认结论：两种展示并存。单笔录入Form：月份是行（一行一个月份）。已填报数据查看：客户+产品为行，月份为列横排，Tab切换不同月。**

🔴 **13. 填报时的数据校验规则** ✅已确认
- 金额/数量：可为0（某些月份无数据），但**不能为负**（>= 0）
- 开票金额与订单金额无关联，各自分别独立填写，无上限/比例限制
- 月度数据与年度汇总无校验关系，各自独立

🟢 **14. 历史数据的只读控制** ✅已确认
- 数据提交后进入审批流，提交人不可编辑
- 当前审批节点的操作人（直线经理→区域经理→最终审批人）可对数据做修改后再提交
- 只有被退回后，退回节点的人才可编辑
- 最终审批人（Frank Tao）只审批不修改，有问题直接退回

🔴 **15. 一个销售在同一周期同一客户同一产品能否产生多条记录** ✅已确认
不可重复，同一（销售+周期+客户+产品+月份）唯一一条记录。批量上传时相同组合覆盖更新。

## §3.3 审批流程（9个问题）

🔴 **16. 审批推进机制：自动还是手动** ✅已确认
通过则自动到上一层（直线经理→区域经理→最终审批人），全程无需手动推送。

🔴 **17. 审批链节点是否一定存在** ✅已确认
若某级审批人在组织架构中不存在（如某销售无直线经理，只有区域总监），审批流自动跳过该级，找到下一个有效审批人继续向上流转。

🔴 **18. 退回层级逻辑** ✅已确认
退回是一层一层往回退（不是跳级退到原点）：最终审批人驳回→区域经理→直线经理→销售。被退回的节点可编辑，修改后重新提交再往上走。

🔴 **19. 退回次数是否有限制** ✅已确认
无次数限制。退回后审批log完整保留，可无限修改重新提交，继续从退回节点往上走。

🔴 **20. Frank Tao的约束是前端还是后端** ✅已确认
后端控制。审批流程配置中每个节点独立配置"是否可修改数据"。最终审批人通常配置为不可修改（只审/退），其他节点按需配置。

🔴 **21. 调整动作后的数据重新提交流程** ✅已确认（与Q18一致）
退回到哪层就从哪层继续往上提交，不从头开始。

🟢 **22. 批量审批能力**
直线经理有10个下属批量提交时，是逐个处理还是支持"批量通过/驳回"？批量操作时调整值怎么应用？

🔴 **23. 审批撤回/撤销** ✅已确认
提交后不可撤回。只能等上一级审批人主动驳回来终止流程。

🔴 **24. 审批历史的数据量和查询性能** ✅已确认（非问题）
数据量不会特别大，记录永久保留。

## §4.1 Dashboard（6个问题）

🔴 **25. Dashboard时间范围定义** ✅已确认（见Q14补充）
趋势图是当前周期内所有月份，还是最近N个月？跨品牌/区域时数据怎么聚合？
→ **确认结论：Dashboard永远显示最新有效数据——有活跃周期时显示当前周期，已全部关闭时显示最近一个有数据的周期。Dashboard不能空着。**

🔴 **26. Dashboard各角色数据范围一致性** ✅已确认
组织架构四层：销售→直线经理→区域总监→总经理。上级可见自己+所有下级数据。直线经理自然也看自己+下属（销售）的数据。

🔴 **27. Dashboard客户维度汇总的口径** ✅已确认（已实现）
卡片同时展示订单金额+开票金额；图表有2个Tab切换（订单金额/开票金额）；按客户维度汇总。

🟢 **28. Dashboard数据统计口径和刷新频率**
只含已审批通过？含审批中？还是可选切换？实时查询还是定时汇总？审批状态变化后Dashboard多久更新？

🟢 **29. 空状态和加载状态**
无数据时显示什么？Loading骨架屏？API报错时友好提示？

🔴 **30. 未提交/草稿数据是否计入Dashboard** ✅已确认（与Q10关联）
销售提交了但经理还没审，直线经理的Dashboard能看到吗？
→ **确认结论：Dashboard和报表只含已审批通过数据。草稿、审批中的数据不计入。**

## §4.2 填报（6个问题）

🔴 **31. 复制上期数据的逻辑** ✅已确认
每年四次预测，每次周期月份范围一致。"复制上期"=复制当前窗口对应月份的去年的预测数据（如27FC1复制26FC1对应月份的数据）。复制后为草稿状态，需重新提交审批。

🔴 **32. 草稿提交后的状态变化** ✅已确认
草稿提交后原记录状态直接变更，不新建记录。审批状态机（可配置）：
1. 空白 → 2. 草稿（已保存）→ 3. 提交至直线经理审批 → 4. 提交至区域总监审批 → 5. 提交至总经理审批 → 6. 审批已完成

🔴 **33. 填报月份动态生成的规则** ✅已确认
月份根据 ForecastPeriod 的 period_start/end 动态生成，一定是连续不间断的月份序列。

🔴 **34. 填报时业绩归属区域和销售大区的自动带出时机** ✅已确认
每次进入填报页实时查 OrgNode，取最新组织架构数据。历史记录保持填报时的值不变。组织架构调整后的数据迁移属后台手工操作，不在系统正常流程内。

🔴 **35. 开票公司和客户的关系** ✅已确认
开票公司和客户没有关联，任何客户可开任何公司的票。

🔴 **36. 填报时的数量精度和汇率处理** ✅已确认
系统不展示单位，金额精确到元。无多币种。

## §5.2 数据权限（5个问题）

🔴 **37. 品牌过滤的强制性和实现层级** ✅已确认
山工多品牌，每个销售只归属一个品牌，每个客户也只归属一个品牌。过滤逻辑：Salesperson.Brand = Customer.Brand，确保销售只能填报自己品牌下的客户。品牌过滤在后端查询层强制执行，不可绕过。

🔴 **38. 直线经理"团队数据"的递归深度** ✅已确认
地歸关系：所有上层可见所有直接或间接下层数据，无递归深度限制。

🟢 **39. 草稿状态下的权限边界**
草稿状态时销售只能看到自己的数据，但提交后经理审批前，直线经理能看到这条数据吗？

🔴 **40. OrgNode树的数据结构** ✅已确认（技术实现）
采用 parent_id 自引用结构。递归深度固定4层、规模有限，递归CTE查询性能无问题。

🔴 **41. 跨部门特别权限和审批链权限的优先级** ✅已确认
取并集。特别权限与审批链权限叠加，不冲突。

## §6 实体和数据库（5个问题）

🔴 **42. 外键约束和级联删除** ✅已确认
所有基础数据（客户/产品/周期）均采用软删除（is_deleted），历史记录保留展示。

🔴 **43. ApprovalRequest和ApprovalHistory的关系** ✅已确认
一个 ApprovalRequest 对应多条 ApprovalHistory（多轮审批/退回每次都记录）。

🔴 **44. 同一用户同一周期同一客户同一产品的记录唯一性** ✅已确认
同一（用户+周期+客户+产品+月份）唯一一条记录，每次提交覆盖。ApprovalHistory仅记录审批时间点、状态、操作人、评语，不记录具体数据值，无法做数据追溯/回滚。

🔴 **45. 软删除的统一拦截** ✅已确认（技术实现）
统一采用 EF Core Global Query Filter，BaseEntity 添加 `Where is_deleted = 0` 自动拦截，无需每个查询点自写。

🔴 **46. 数据一致性的事务边界** — 不需要处理（非blocking，无实际影响）

## §9 M365 SSO（4个问题）

🔴 **47. 首次SSO登录的用户处理** ✅已确认
返回错误，不自动建账号。用户须管理员提前在系统建好才能登录。

🟢 **48. SSO Token有效期和刷新策略**
Access Token多久过期？过期后自动刷新还是重新登录？Refresh Token存哪（HttpOnly Cookie还是localStorage）？

🔴 **49. 多设备登录处理** ✅已确认
不可多设备同时登录，相互排斥。一处登录，其他设备自动退出。

🔴 **50. 本地开发环境的登录切换** ✅已确认（技术实现）
开发环境用 JWT 登录，生产环境强制 SSO。靠 `ASPNETCORE_ENVIRONMENT=Development` 环境变量区分。

## §10 产品管理（4个问题）

🔴 **51. 产品5级联动的UI级联行为** ✅已确认
适用范围：产品录入 + 预测数据填报 + 预测数据查询，统一规则。任一级变动时，其下级清空强制重选。PRD中某些级别之间有断层、不联动。SubPS4（最后一级）型号数量大，采用输入模糊搜索+下拉选择的方式。

支持模糊搜索（左右模糊匹配），可按俗称搜到规范名称。

🔴 **53. L1产品线的初始化** ✅已确认
上线时统一提供文件导入，各产品线分开文件。可分批导入，支持增量更新。

🔴 **54. 产品数据初始化方式** ✅已确认
全量覆盖。流水号（产品编码）不变。产品编码规则：各层级编码相加组成十余位编码（详见PRD）。
